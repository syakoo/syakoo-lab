# ルール・スキルの提案

## いつ作るか

以下のいずれかに該当する場合に提案を検討する。

- **反復**: 同じ判断・手順を 2 回以上繰り返した
- **暗黙知**: 会話の中でしか伝わらない知識がある
- **落とし穴**: 他のセッションで同じミスをしそうなパターンがある

1 回しか遭遇していないものは様子を見る。

## ルールとスキルの使い分け

| 作るもの | 置き場 | 適用 | 向いているもの |
|---------|--------|------|--------------|
| ルール (`.mdc`) | `.cursor/rules/` | 常時 or ファイルパターン | 短い方針・制約・コーディング規約 |
| スキル (`SKILL.md`) | `.cursor/skills/` | オンデマンド | 手順・ワークフロー・判断ロジック |

迷ったら「50 行以内で書けるか」で判断する。書けるならルール、書けないならスキル。

## ワークフロー

### 1. 本実装を先に完了する

ルール・スキルの作成は本実装の PR 作成後に行う。本実装 PR とは **必ず別 PR** にする。

### 2. 重複確認

リポジトリ内（`.cursor/`）と個人スキル（`~/.cursor/skills/`）の両方を確認する。完全に既存でカバーされていれば作成不要の旨をユーザーに報告して終了、部分的に不足していれば既存の修正で対応する。

### 3. 提案の合意

重複がないことを確認した上で、ユーザーに「このパターンをルール/スキル化したい」と伝え、合意を得る。

伝える内容:
- **何を**: ルールかスキルか
- **なぜ**: どういう場面で繰り返し発生したか
- **概要**: どんな内容になるか 1〜2 行

### 4. 作成

- ルールの場合: `create-rule` スキルの規約に従う
- スキルの場合: `create-skill` スキルの規約に従う

### 5. チューニング

`empirical-prompt-tuning` スキルを使い、コンテキストなしの fresh executor で検証する。チューニングが収束するまで PR は出さない。

同セッションでチューニングまで完了できない場合は、ユーザーにその旨を伝え、次のセッションに回してよい。その際、ドラフトをブランチにコミット・プッシュしておき、次セッションで引き継げるようにする。

### 6. PR 作成

`create-pull-request` スキルに従い PR を作成する。PR 本文の「課題」セクションに以下を明記する:

- **動機**: なぜこのルール/スキルが必要か
- **適用場面**: どういう作業で使われるか
- **根拠**: どのセッション・作業で必要性を認識したか

## 制約

- 本実装の PR にルール・スキルの変更を混ぜない
- ユーザーの合意なしに作成を開始しない
- `empirical-prompt-tuning` によるチューニングを省略しない
- 既存のルール・スキルと重複する内容を作らない（ステップ 2 の重複確認を省略しない）
